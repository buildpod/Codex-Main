# BuildPodFactory Project Context

Use this file to brief ChatGPT or another reviewer before asking for architectural, product, or implementation feedback.

## Project purpose

BuildPodFactory is a private, local, governance-first AI software factory. Its purpose is to turn a human-selected software product idea into a controlled product output: a git repository, validation/evidence package, README, and release decision trail.

The factory itself is the durable operating system: agents, gates, evidence, approvals, learning, and release discipline. Apps generated by the factory are downstream artifacts. Jarvis is the local command center/prototype for operating the factory; it is not the product to sell and should not be deployed as a public product.

The current strategic wedge is regulated or compliance-heavy product building, especially pharma/GxP-style workflows. Phase 0 design is complete. Phase 1 has started with small carry-over filings, but the real factory engine is not implemented yet.

## Tech stack

Implemented now:

- Runtime: Node.js with native ES modules.
- Package manager: pnpm.
- Servers: Node `http` module, no Express.
- Frontend: plain HTML, CSS, and browser JavaScript.
- State: local JSON and Markdown files in the repo.
- Evidence/design docs: Markdown plus JSON schemas/records.
- Git: local version control, with a GitHub remote configured but not pushed in recent Phase 0/Phase 1 work.

Current local apps:

- Jarvis runs on `127.0.0.1:4173` by default.
- PlayPatch Kids runs on `127.0.0.1:4273` by default.

Planned but not implemented as the real factory engine:

- Direct Anthropic SDK use with a pluggable adapter for Anthropic, Ollama, DeepSeek, and OpenAI.
- Master Moderator orchestration with budgets, model routing, file locks, job dependencies, and evidence logging.
- Hash-chained evidence ledger integrated with agent work.
- Internal pre-run prediction module for tokens, cost, and wall-clock time.

## How to run it

From the project root:

```bash
pnpm install
```

Start Jarvis:

```bash
pnpm jarvis
```

Open:

```text
http://localhost:4173
```

Safer Jarvis development start, which avoids duplicate healthy servers:

```bash
pnpm jarvis:dev
```

Run Jarvis checks:

```bash
pnpm jarvis:check
pnpm jarvis:health
pnpm jarvis:smoke
```

Start PlayPatch Kids:

```bash
pnpm playpatch
```

Open:

```text
http://localhost:4273
```

Run PlayPatch checks:

```bash
pnpm playpatch:check
pnpm playpatch:smoke
```

Run factory dry-run harnesses:

```bash
pnpm factory:dry-run
pnpm factory:dry-run:blocker
```

## Known issues

- The real Phase 1/Phase 2 factory engine is not built yet. The current repo has strong paper design plus Jarvis and dry-run scaffolding.
- Jarvis is a prototype command center, not a real autonomous multi-agent backend yet.
- Jarvis/PlayPatch/runtime files are currently dirty in git; review should distinguish committed handoff docs from unstaged prototype changes.
- The approved Phase 0 roster has 11 agents with QA/Test and Security/Compliance split. Some older Jarvis runtime labels may still refer to a combined QA/Security role.
- Evidence ledger design is approved, but the full hash-chained implementation is not wired into all runtime operations.
- LLM adapter and real provider calls are not implemented as the factory engine.
- Pre-run prediction is deferred. Tokencostscope is not installed and should not be treated as a dependency.
- Retention period, cryptographic signature standard, and exact JSONL evidence storage layout are still open design questions.
- `tree` is not installed in the local shell, so structure snapshots may need `find` unless the reviewer installs `tree`.
- Do not deploy Jarvis. It is local factory infrastructure, not a customer app.
